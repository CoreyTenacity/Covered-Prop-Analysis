# Covered. API key plan

The app continues to work with mock data and no keys. Add providers in the product roadmap order, and keep every secret server-side unless its name explicitly begins with `NEXT_PUBLIC_`.

## Recommended lean stack

1. **The Odds API — `THE_ODDS_API_KEY`**
   Board lines and sportsbook consensus for MLB, WNBA, NBA, and NFL. Its current DFS coverage includes PrizePicks, and its standard player-prop markets provide comparison lines.
2. **SportsDataIO — `SPORTSDATAIO_API_KEY`**
   Primary player stats, projections, injuries, lineups, schedules, and eventual result grading. A real-time production feed requires a commercial plan; its self-serve Discovery Lab is delayed and does not cover WNBA or Tennis.
3. **Supabase — URL, publishable key, and secret key**
   Pick persistence, history, cached opportunities, and calibration. The publishable key is client-safe with Row Level Security; the secret key is server-only.
4. **OpenRouter — `OPENROUTER_API_KEY`**
   Optional explanations only. The deterministic engine—not AI—scores and ranks picks.
5. **Open-Meteo — no key for non-commercial use**
   MLB/NFL weather. Commercial reserved resources use a customer API key.
6. **Vercel — `CRON_SECRET` generated by us**
   Scheduled refresh protection. This is a random application secret, not a purchased API key.

## Alternatives, not additions

- **API-Sports — `API_SPORTS_KEY`**: lower-cost stats alternative, but verify endpoint-level WNBA and Tennis coverage before choosing it as the one primary stats feed.
- **Sportradar — `SPORTRADAR_API_KEY`**: enterprise-grade alternative for deep official sports feeds. Do not buy this alongside SportsDataIO for the MVP.

## Official injury reports

NBA, WNBA, and NFL injury-report URLs can be wired as optional best-effort enrichment sources through `NBA_INJURY_REPORT_URL`, `WNBA_INJURY_REPORT_URL`, and `NFL_INJURY_REPORT_URL`. These should point at a machine-readable page or endpoint if possible. If the page becomes unavailable or its structure changes, Covered. will keep the board alive and mark the injury context as unavailable instead of failing the refresh.

## Big Balls Sports Data configuration

Add `BBS_API_KEY` to `.env.local`. It is a server-side secret and must never use a `NEXT_PUBLIC_` prefix. Requests authenticate with `Authorization: Bearer <key>`.

- The current free plan advertises 1,000 requests per day, or 2,000 with GitHub signup, without a credit card.
- NBA injury reports are currently listed as available on the free basketball API.
- Baseball and American-football injury coverage are currently marked as coming soon. Covered. must not treat those leagues as having live injury coverage until the provider exposes and verifies those feeds.
- If you need deeper MLB player logs or Statcast-style detail, the current MLB providers are good for match discovery, lineups, identity, and some stats, but not a complete substitute for the MLB Stats API/Statcast layer.

## Official MLB and NBA stats

The app now uses official, no-key enrichment paths for deeper logs:

- MLB: the official MLB Stats API docs at [docs.statsapi.mlb.com](https://docs.statsapi.mlb.com) are the source of record for player logs and season context.
- NBA: the app uses nba_api-compatible NBA.com endpoints from the free/open-source [nba_api](https://github.com/swar/nba_api) ecosystem as the reference pattern.
- WNBA: not assumed yet. We can verify a supported league path before relying on it.

These feeds are optional, cached, and strictly server-side. If they fail, the board keeps working and the pick simply carries less context.

## Statcast / Baseball Savant

For deeper MLB player logs and pitch-level context, Covered. also uses the free Baseball Savant / Statcast CSV path. It is optional, cached, and server-only. If Savant changes shape or returns no rows, the board keeps working and the app simply loses that extra pitch-level detail.

After changing `.env.local`, restart the local app. For the public Vercel site, add the same variable names under Project Settings → Environment Variables and redeploy. Do not paste secret values into source files or commit `.env.local`.

Adding a key only configures access; it does not activate requests. Provider adapters will enforce request limits, freshness timestamps, and stale-data warnings before these feeds affect a Covered Score.

## Shared quota and refresh policy

Browser page loads and refreshes must never call a third-party sports provider. A scheduled server job refreshes shared Supabase cache records, and every user reads the same cached snapshot.

- Highlightly: 100 published daily requests; Covered hard-stops at 80 by default.
- API-Sports: 100 published daily requests on the free plan; Covered hard-stops at 80 by default.
- Big Balls Data: 1,000 published daily requests on the free plan; Covered hard-stops at 800 by default.
- The Odds API: 500 monthly credits divided by the calendar days and five sports permits three complete five-sport refreshes per day (15 credits). Those runs are spaced eight hours apart. In a 31-day month this uses at most 465 credits, leaving 35 in reserve; the monthly counter remains the final hard stop.

The safety budgets can be lowered with environment variables. When a hard budget is reached, the backend serves the last cached result with a delayed-data notice instead of spending more quota. Live games receive the shortest refresh interval; scheduled games refresh less often; completed games are cached for hours and later rechecked only for grading corrections.

The action-by-action usage, storage, and retention design is documented in [API_USAGE_AND_STORAGE.md](./API_USAGE_AND_STORAGE.md).

## SportsGameOdds configuration

Add `SPORTS_GAME_ODDS_API_KEY` to `.env.local`. Covered. will access API v2 from the server and cache shared results. Because this provider meters returned events/objects, `SPORTS_GAME_ODDS_DAILY_OBJECT_BUDGET` defaults to `0`: no request is authorized until the allowance shown in the account is known and entered. Page views and manual analyzer choices never call it.

- `SPORTS_GAME_ODDS_BASE_URL` defaults to `https://api.sportsgameodds.com/api/v2`.
- `SPORTS_GAME_ODDS_ODDS_PATH` defaults to `/odds`.
- `SPORTS_GAME_ODDS_TENNIS_ODDS_PATH` defaults to `/odds`.
- `SPORTS_GAME_ODDS_TENNIS_SPORT_KEYS` lets the server try multiple tennis league codes, in order, until it finds live rows.
- If their docs call for a different auth header, set `SPORTS_GAME_ODDS_AUTH_HEADER` and `SPORTS_GAME_ODDS_AUTH_SCHEME` instead of changing code.
- The adapter is player-prop-only by default so it stays aligned with the board rules.
- Tennis uses SportsGameOdds as the first live prop provider when configured; MLB/WNBA/NBA/NFL still prefer The Odds API and can use SportsGameOdds as supplemental coverage.

## SharpAPI sports configuration

Add `SHARPAPI_KEY` to `.env.local`. Server requests authenticate with the `X-API-Key` header. Covered uses SharpAPI for player props only and rotates between MLB and WNBA pages. `SHARPAPI_DAILY_REQUEST_BUDGET` defaults to `0`, which means no daily cap inside Covered, while `SHARPAPI_REQUESTS_PER_REFRESH` defaults to 12. With the two-minute refresh schedule this is still below the provider's 12-requests-per-minute rate ceiling if the bursts stay short and the cached cursor keeps advancing. Stored rows are deduplicated by event, player, market, direction, line, and sportsbook before normalization.

## Highlightly MLB configuration

Add `HIGHLIGHTLY_MLB_API_KEY` to `.env.local`. The direct Highlightly integration uses `https://baseball.highlightly.net` and sends the key through the `x-rapidapi-key` request header. `HIGHLIGHTLY_MLB_BASE_URL` is already configured with that direct base URL.

- The current Basic plan advertises 100 requests per day without a credit card.
- The key and base URL must remain server-side.
- If the subscription was created through RapidAPI instead of Highlightly directly, the provider adapter will also need the RapidAPI host. Do not add that host for a direct Highlightly account.
- Before injury data changes a Covered Score, the adapter must verify the endpoint response, capture its retrieval time, and treat missing or stale results as unavailable.

## Player images

Covered. uses generated avatars instead of real player photos for now. That keeps the board visually complete without introducing a headshot dependency that can break the layout when a provider omits images.

## Important board-line note

There is no separate official PrizePicks developer key in the current project plan. The practical approved-provider route is The Odds API's `prizepicks` DFS bookmaker feed. Manual entry remains the fallback and prevents the product from depending on scraping.
